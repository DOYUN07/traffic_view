## 1. 개요 (Overview)

**TrafficView**는 보행자가 횡단보도 신호 타이밍에 맞춰 **최적의 걸음 속도**로 이동하도록 안내하는 모바일 웹 앱이다.
일반 길찾기 앱이 "어디로 갈지"를 알려준다면, TrafficView는 **"신호에 걸리지 않게 어떻게 걸을지"**를 알려준다.

> 한 줄 정의: *다음 초록불에 딱 맞춰 도착하도록 걸음 속도를 실시간으로 코칭하는 보행 내비게이션.*

---

## 2. 문제 정의 (Problem Statement)

- 보행자는 횡단보도마다 **빨간불 대기**로 이동 시간을 예측하기 어렵고, 불필요하게 멈춰 선다.
- 신호가 곧 바뀌는데 무리하게 뛰다가 **사고 위험**에 노출된다.
- 시각장애인·고령자 등 교통약자는 신호 상태와 잔여 시간을 **즉시 인지하기 어렵다**.

기존 내비게이션은 경로·거리·소요시간은 주지만, **실시간 신호 상태를 반영한 보행 속도 안내**는 제공하지 않는다.

---

## 3. 목표 (Goals)

### 3.1 제품 목표
- 경로상의 보행 신호를 실시간으로 파악해 **대기 시간을 최소화**하는 걸음 속도를 안내한다.
- 신호 상태·잔여 시간·건너기 시점을 **시각·음성·진동**으로 명확히 전달한다.
- 교통약자가 안전하게 횡단보도를 건널 수 있도록 **접근성**을 1급 기능으로 제공한다.

### 3.2 비목표 (Non-Goals)
- 차량 내비게이션 / 대중교통 환승 안내는 범위 밖.
- 전국 모든 교차로 신호 커버리지 보장(공공데이터 제공 지역에 한정).
- 자체 신호 예측 모델 개발(공공 API의 잔여시간을 그대로 사용).

---

## 4. 타깃 사용자 (Target Users)

| 페르소나 | 니즈 |
|----------|------|
| 출퇴근 도보 이동자 | 신호 대기 없이 빠르게, 뛰지 않고 이동 |
| 시각장애인 | 신호 상태·건너기 시점의 명확한 음성·진동 안내 |
| 고령 보행자 | 큰 글씨·고대비 화면, 무리한 횡단 방지 |
| 야간 보행자 | 눈부심 적은 화면, 또렷한 신호 표시 |

## 5. 핵심 기능 (Features)

### 5.1 보행 속도 추천 (핵심 차별점)
- 다음 신호까지의 거리·현재 신호 위상을 바탕으로 **초록불 도착 가능 속도**를 계산.
- `recommendSpeed()`: 향후 초록불 구간(green window)들 중 편안한 속도(기준 1.25 m/s)에 가장 가까운 도착 속도를 선택.
- 결과를 "유지 / 조금 빠르게 / 천천히" 힌트와 권장 km/h로 표시.

### 5.2 실시간 신호 연동
- 공공데이터포털(`B551982/rti/tl_drct_info`) 보행신호 잔여시간 API를 5초 주기로 폴링.
- CORS 우회용 Cloudflare Worker 프록시 경유.
- 연동 실패 시 **시뮬레이션 모드로 자동 전환**(키 없이도 데모 동작).

### 5.3 경로 안내 화면 흐름
```
검색(SearchScreen) → 경로 요약(SummaryScreen) → 안내(NavScreen) → 도착(ArrivedScreen)
```
- 다음 신호 배너(큰 카운트다운), 권장 속도 게이지, 진행률, 남은 신호 목록.
- 신호 탭 시 상세 시트(초록/빨강 시간, 전체 주기 타임라인).

### 5.4 접근성 (Accessibility)
- **고대비 모드**: 순수 흑백 + 진한 신호색, 큰 글씨(1.16배).
- **음성 안내**: Web Speech API로 신호 상태·건너기 시점을 한국어로 발화(접근성 모드에서 상시 ON).
- **진동 피드백**: 신호 안내 시 `navigator.vibrate`.
- **야간 모드**: 다크 테마.

### 5.5 실제 지도 + GPS (Phase 1, 키 연동 시)
- 카카오 지도 SDK로 실제 지도·경로·신호 마커 렌더.
- 브라우저 Geolocation으로 실제 내 위치 표시.
- 키 미설정 시 SVG 시뮬레이션 지도로 폴백.

|----|----------|----------|

| FR-2 | 다음 신호 도착 기준 권장 보행 속도 계산·표시 | P0 |
| FR-3 | 실시간 신호 API 연동 및 실패 시 폴백 | P0 |

### 6.2 비기능 요구사항
- **성능**: 신호 폴링 5초, UI 60fps 유지.
- **호환성**: 모바일 웹(반응형, 390×844 기준), 최신 크롬/사파리.
- **보안**: API 키는 `config.local.js`(gitignore)로 분리, 저장소 미포함.
- **오프라인 내성**: API 실패 시에도 시뮬레이션으로 기능 유지.

## 8. 기술 아키텍처 (Technical Architecture)

### 현재
- React 18 + Babel Standalone(빌드리스, 브라우저 런타임 트랜스파일).
- 단일 진입점 `traffic_view.html` + 모듈형 `.jsx`.

| 파일 | 역할 |
|------|------|
| `data.jsx` | 경로/신호 데이터, 타이밍·속도 로직, geo 좌표 |
| `signal_api.jsx` | 공공데이터 실시간 신호 연동 |
| `map.jsx` | SVG 시뮬레이션 지도 |
| `kakao_map.jsx` | 카카오 실지도 + GPS (키 연동 시) |
| `app.jsx` | 앱 셸, 화면 전환, 설정 |
| `proxy/cloudflare-worker.js` | CORS 우회 프록시 |

### 목표 (프로덕션)
- **Vite + React** 번들 빌드로 전환.
- 지도: 카카오 지도 SDK · 경로: TMap 보행자경로 API · 위치: Geolocation.
- 배포: PWA(홈화면 설치) + HTTPS(Vercel/GitHub Pages).

### 알려진 리스크
- **신호 매칭 난제(Phase 3)**: 공공 API 응답에 교차로 좌표가 없어, `crsrdId`와 실제 횡단보도 위치를 잇는 별도 데이터셋·매칭 로직이 필요하다. 제품 성패의 핵심.
- **신호 데이터 커버리지**: 보행신호 실데이터 제공 지역이 제한적(서울 등).
- **API 키 발급 의존**: 카카오·TMap 키는 사용자 계정 발급·도메인 등록 필요.
